## Problem Statement

The Stanford Meta-Harness paper (March 2026) showed that searching over harness code with access to raw execution traces dramatically outperforms hand-designed harnesses. But the approach costs ~$500 per run (88M tokens) because it uses a full coding agent (Claude Code) to propose Python harness variants. PTC-Runner has a unique structural advantage: harnesses are small PTC-Lisp programs in BEAM sandboxes with millisecond evaluation, structured trace data, and existing LLM-as-tool infrastructure. The question: can PTC-Runner's architecture make Meta-Harness-style optimization 100x cheaper?

This supersedes the M0 (Failure Boundary Mining) plan, which required 7 new tools, an orchestrator, and trace storage extensions before anything could run. The new approach validates the same core hypothesis — can we automatically improve agent performance? — by reusing existing infrastructure.

## What Makes This Cool

A system that evolves its own agent strategies and discovers when to think (call LLM) vs when to compute (run PTC-Lisp). The fitness function penalizes LLM calls, so evolution naturally discovers the cheapest path to solving problems. Harnesses aren't opaque Python files — they're structured PTC-Lisp programs you can analyze, diff, mutate at the AST level, and replay in milliseconds. The Meta-Harness paper's key insight (raw traces > scores > summaries) comes for free because PTC-Runner already produces structured trace data.

## Constraints

- Must reuse existing SubAgent.run/2, Tracer, TraceLog, TurnAnalysis
- Minimal new infrastructure for Phase A (HarnessSpec struct + eval + mutators only)
- LLM calls allowed everywhere but with explicit fitness penalty
- PtcRunner is 0.x — delete freely, don't add compatibility shims
- Sandbox limits: 1s timeout, 10MB memory per execution
- Budget-conscious: minimize LLM calls during development and experimentation

## Premises

1. **M0 is too heavy as a starting point** — a Meta-Harness-inspired loop validates the same idea (automatic agent improvement) with less infrastructure by reusing SubAgent.run/2 and existing traces.
2. **The "harness" to optimize is the SubAgent configuration** — system prompt sections + prelude functions + parameters (max_turns, temperature, tool budget). Not arbitrary code.
3. **Existing trace infrastructure is the feedback signal** — Tracer, TraceLog, and TurnAnalysis already produce rich enough data. No new tools needed.
4. **LLM calls are allowed but costly** — LLM calls during both proposal and execution carry a fitness penalty. Evolution pressures the system to minimize LLM usage when cheaper symbolic alternatives suffice. This enables "distillation through evolution."

## Cross-Model Perspective

Codex proposed a "co-evolving harness league" where a second population turns failure traces into adversarial micro-benchmarks. Every new failure becomes a replayable PTC-Lisp counterexample, preventing benchmark overfitting. This is the 10x vision (Approach C) but requires the basic loop (A+B) to work first.

Codex identified DSPy as 50% of the way there conceptually (treat LM program as optimizable object), but the 50% to build is PTC-Runner-specific: HarnessSpec as Elixir data, AST-aware mutation for PTC-Lisp, SubAgent.run/2 as evaluator, fitness from traces with LLM cost penalties.

Codex's weekend build spec closely matched Approach A: HarnessSpec with 5 knobs, 10 tasks, 5-6 dumb mutators, 32-64 candidates, hill-climbing, output winning harness diff plus before/after traces.

## Approaches Considered

### Approach A: "Weekend Loop" — Minimal Viable Meta-Harness

Define HarnessSpec (6 knobs mapping to real Definition fields: prompt_sections, prelude_source, max_turns, turn_budget, max_tool_calls, retry_turns). Build MetaHarness.eval/3 that runs one spec through SubAgent.run/2 against 10 test cases with N=3 runs each, returns fitness = solve_rate - token_penalty - turn_penalty. 6 simple mutators (add/remove/swap prompt sections, toggle prelude, nudge params). (mu+lambda) evolution over 8 generations. Log everything to disk.

- **Effort:** S (1-2 sessions)
- **Risk:** Low
- **Pros:** Fastest to learn from. Validates core loop. Minimal new code (4 small modules).
- **Cons:** No trace-based feedback (just scores). No LLM proposer. Mutators are dumb.
- **Reuses:** SubAgent.run/2, Definition struct fields, existing test cases + constraints, TurnAnalysis, Tracer

### Approach B: "Trace-Aware Meta-Harness" — The Full Cheap Version

Same as A, but the proposer (a single cheap LLM call per iteration) gets structured trace data: failure clusters, error types, tool call patterns, TurnAnalysis metrics. The LLM proposes targeted mutations based on WHY harnesses fail. Filesystem-based logging of all candidates + traces. Fitness includes LLM cost penalty for both proposer calls and execution calls.

- **Effort:** M (3-4 sessions)
- **Risk:** Medium
- **Pros:** Tests the Meta-Harness paper's key finding (traces > scores). More likely to find real improvements.
- **Cons:** More infrastructure. Need to decide trace format for the proposer.
- **Reuses:** SubAgent.run/2, test cases, Tracer, TraceLog, TurnAnalysis

### Approach C: "Harness League" — Co-Evolving Harnesses + Benchmarks (future)

Two populations: harness variants AND adversarial test generators. Failures from one generation become test cases for the next. AST-level mutation of PTC-Lisp preludes. LLM used for creative mutations only, with annealing cost.

- **Effort:** L-XL
- **Risk:** High
- **Pros:** Solves fixed-benchmark saturation. Most interesting research artifact.
- **Cons:** Too much to build at once. Requires A+B to work first.

## Recommended Approach

**A first, then B.** Start with the weekend loop (A) to prove the optimization mechanism works with dumb mutators and score-only feedback. Once we see real harness variants competing, add trace-aware LLM proposing (B) to test whether the Meta-Harness paper's key insight (raw traces are essential) holds in PTC-Runner's structured world.

### Phase A: HarnessSpec + Eval + Mutate + Select

#### HarnessSpec maps to SubAgent.Definition

The spec is a thin wrapper over `SubAgent.Definition` fields that already exist:

```elixir
# The spec — what we're optimizing
%HarnessSpec{
  id: "harness-gen0-v1",
  parent_id: nil,
  generation: 0,

  # Maps to Definition fields:
  # prompt_sections are appended to the base Definition's system_prompt (inherited)
  prompt_sections: ["Break the problem into steps"],
  prelude_source: [],             # PTC-Lisp source strings, injected via initial memory
  max_turns: 3,                   # Definition.max_turns (exists, default 5)
  turn_budget: 20,                # Definition.turn_budget (exists, default 20)
  max_tool_calls: nil,            # Definition.max_tool_calls (exists, default nil)
  retry_turns: 0                  # Definition.retry_turns (exists, default 0)
}
```

**NOTE: temperature is NOT a knob.** The Definition struct has no temperature field, and
the LLM adapter layer doesn't expose it. Removed from the spec. We optimize what
SubAgent.Definition actually supports.

**NOTE: prelude injection.** PTC-Runner's `Namespace.User` renders user-defined functions
from the memory map (shown as `;; === user/ (your prelude) ===`). To inject prelude
functions, we parse each `prelude_source` string into PTC-Lisp, evaluate it to produce
closures, and add them to the initial memory map passed to SubAgent.run/2. This requires
a small helper (`HarnessSpec.to_initial_memory/1`) but no changes to SubAgent internals.
If a prelude source string fails to parse, it is silently excluded from the memory map
and the parse error is logged in the harness evaluation results.

#### Seed Harness (baseline)

The seed harness uses the same configuration as `PtcDemo.LispTestRunner` with defaults:

```elixir
@seed %HarnessSpec{
  id: "seed",
  prompt_sections: [],            # no extra instructions
  prelude_source: [],             # no prelude functions
  max_turns: 5,                   # Definition default
  turn_budget: 20,                # Definition default
  max_tool_calls: nil,            # unlimited
  retry_turns: 0                  # no retries
}
```

All harness variants are compared against the seed. The seed's fitness is the baseline.

#### How "solved" is determined

Test cases in `PtcDemo.TestRunner.TestCase` already define `:expect` (return type) and
`:constraint` (e.g., `{:eq, 500}`, `{:between, 1, 100}`, `{:gte_length, 3}`). The
existing `LispTestRunner` uses these to determine pass/fail. `MetaHarness.eval/2` reuses
the same validation: a problem is "solved" when SubAgent.run/2 returns a value matching
both the expected type and constraint.

#### Error handling

When a harness variant causes SubAgent.run/2 to crash, timeout, or exceed memory:
- The problem counts as **not solved** (solve_rate denominator stays the same)
- Token usage from the failed run is still counted toward fitness penalty
- The error type is logged in the trace (already handled by Tracer)
- No candidate is excluded — bad harnesses get low fitness, which is selection pressure

#### Evaluation

```
fitness(spec, problems) =
  solve_rate(spec, problems)
  - lambda_tokens * total_tokens / 1000
  - lambda_turns * avg_turns / max_turns
```

Starting lambdas: `lambda_tokens = 0.01`, `lambda_turns = 0.05`. Fixed for Phase A.

**Variance mitigation:** Each problem is run N=3 times per harness spec. solve_rate is
the fraction of (problem, run) pairs that pass. With 10 problems * 3 runs = 30 trials,
a harness scoring 80% vs 70% differs by 3 successes. For Phase A (learning, not
publishing), this is sufficient signal for hill-climbing. Phase B can increase N or add
paired comparison tests if needed.

#### Mutators (no LLM needed)

1. Add a prompt section from the pool (appended to system_prompt)
2. Remove a random prompt section
3. Swap two prompt sections (order in system_prompt matters)
4. Toggle a prelude function source on/off
5. Nudge max_turns +/- 1 (clamped to 1..10)
6. Nudge turn_budget +/- 5 (clamped to 5..50)

**Prompt section pool** — 10-15 hand-written instruction fragments, e.g.:
- "Break the problem into independent subtasks before solving"
- "Verify your answer by re-reading the question"
- "Use the simplest approach that could work"
- "If a tool call fails, try a different approach"
- (extracted from existing benchmark configs + common agent strategies)

**Prelude function pool** — 3-5 small PTC-Lisp utility functions, e.g.:
- `(defn safe-get [m k default] (if (contains? m k) (get m k) default))`
- `(defn count-where [pred coll] (count (filter pred coll)))`
- (kept small for Phase A; Phase B's LLM proposer can generate new ones)

#### Selection

(mu+lambda) with mu=4, lambda=8. Each generation: 4 parents produce 8 children
(1 random mutation each). Evaluate all 12, keep top 4 as next generation's parents.
Run for 8 generations = 8 * 12 * 30 = 2,880 LLM calls total.

#### Cost estimate (revised)

Multi-turn SubAgent calls average ~2,000-3,000 tokens/run (system prompt + tool calls +
responses). At 2,880 runs * 2,500 tokens avg = 7.2M tokens. With haiku at ~$0.25/1M
input + $1.25/1M output (estimate ~60% input, 40% output): ~$4.70 total. Under $5.

#### Logging

Each candidate writes to disk:
```
meta_harness_log/
  gen-000/
    harness-seed.json      # the harness spec
    harness-seed-results.json   # scores, solve rate, token usage per problem
    harness-seed-traces/        # full traces per problem*run (existing format)
    harness-v1.json
    harness-v1-results.json
    harness-v1-traces/
    ...
  gen-001/
    ...
  summary.json             # best fitness per generation (for plotting)
```

#### Module placement

New modules under `lib/ptc_runner/meta/`:
- `harness_spec.ex` — HarnessSpec struct + `to_definition/2` + `to_initial_memory/1`
- `evaluator.ex` — `eval/3` runs spec against problems, returns fitness + traces
- `mutators.ex` — pure functions: `mutate/2` applies random mutation to spec
- `loop.ex` — (mu+lambda) selection loop with logging

This follows the existing `meta/` namespace hinted at in the M0 plan.

### Phase B: Trace-Aware LLM Proposer

Replace dumb mutators with a cheap LLM call that receives:
- Current best harness spec
- Structured failure summary: which problems failed, error types, TurnAnalysis metrics
- 2-3 representative failure traces (tool calls, outputs, errors)
- The scoring formula and current scores

The LLM proposes a new HarnessSpec (or a diff to the current one). The LLM call's token cost is added to the fitness penalty for that generation.

This tests whether trace-aware proposing finds better harnesses than dumb mutation, and at what cost.

## Open Questions

1. **Problem subset for Phase A** — which 10 of the 55 tests to use? 5 from the original 13 (high baseline) + 5 from the M0-clustered set (lower baseline). The 5 M0 clusters are: A=string extraction, B=date arithmetic, C=nested aggregation, D=set operations, E=conditional logic. Pick 1 from each cluster.
2. **LLM model for evaluation** — the model is configured via the LLM callback passed to SubAgent.run/2 (part of the base Definition). The evaluator uses whatever LLM the user configures. For cost control, recommend haiku for Phase A; validate winners on a stronger model.
3. **Prelude injection mechanics** — need to verify that parsing PTC-Lisp source strings into closures and injecting into initial memory works cleanly with the existing SubAgent flow. Integration test in Next Steps #2.

## Failure Criteria

If no harness variant beats the seed by >5% solve_rate after 4 generations, stop and reassess. Possible causes: mutators too dumb, search space too small, test problems too easy/hard, or the mechanism doesn't work at this scale. Debug by inspecting traces of near-miss variants before concluding the approach failed.

## Success Criteria

1. Phase A produces at least one harness variant that measurably outperforms the seed harness on the test subset
2. The optimization loop runs end-to-end without manual intervention
3. The logging is complete enough to understand WHY a harness won (trace diffs)
4. Phase B's trace-aware proposer finds improvements that dumb mutators miss
5. Total cost of a full optimization run stays under $5

This is a library feature within PTC-Runner. Distribution via Hex package (existing pipeline). No new artifacts to distribute.
